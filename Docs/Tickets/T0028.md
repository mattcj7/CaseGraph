# Ticket: T0028 - Reports startup null-safety hotfix

## Goal
Fix the startup crash caused by `ReportsViewModel` throwing a `NullReferenceException` during construction, and add regression coverage so Reports viewmodel initialization cannot crash app startup again.

## Context
After recent Reports/Incident Window work, the app now crashes on startup while resolving `ReportsViewModel`. The crash occurs because `SelectedSubjectKind` triggers change-handler logic during construction before all state used by `NotifyStateChanged()` is initialized. This is a stability hotfix and should be kept narrow, safe, and well-tested.

## Scope
### In scope
- Fix `ReportsViewModel` construction so it cannot throw during startup.
- Prevent property-change side effects from running against partially initialized state.
- Harden `NotifyStateChanged()` with targeted null-safety for members accessed during early lifecycle transitions.
- Preserve normal Reports behavior after initialization completes.
- Add deterministic regression tests for:
  - viewmodel construction does not throw
  - any existing DI/viewmodel resolution pattern if available

### Out of scope
- Broad Reports page refactor
- New Reports features or workflow changes
- Navigation redesign
- Unrelated startup/performance work

## Acceptance Criteria (Testable)
- [ ] App no longer crashes on startup due to `ReportsViewModel`.
- [ ] Constructing `ReportsViewModel` with representative mocked dependencies does not throw.
- [ ] If there is an existing DI/startup resolution test pattern, resolving `ReportsViewModel` through that pattern does not throw.
- [ ] Reports page still behaves normally after initialization.
- [ ] `NotifyStateChanged()` no longer dereferences uninitialized members during startup.
- [ ] `dotnet test` passes.
