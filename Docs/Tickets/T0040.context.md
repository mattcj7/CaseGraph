# T0040 Context - Entity Profile Viewer Navigation for People and Organizations

## Why this ticket exists
The current People / Targets and Organizations UI proves the base system is working, but the UX still feels like an editor shell instead of an investigator workflow. Records exist, but they are not yet acting like navigable case objects.

The biggest issues to solve now:
- People / Targets and Organizations should open into dedicated profile views.
- Double-click should drill into the selected person or organization.
- People / Targets search/list results should show organization affiliations.
- Blank fields should have persistent labels instead of relying on hover text/tooltips.
- The UI should feel readable and investigation-oriented even before deeper analytics exist.

This ticket should make the app feel much more like a real investigative workspace without taking on too much scope.

## Product intent
This is a usability/navigation ticket, not a major data model or workflow expansion ticket.

The MVP goal is:
- investigator sees a person or org in a list
- investigator opens it immediately
- investigator can understand who/what it is from one screen
- investigator can move between linked people and organizations naturally

The current edit/create screens remain important, but this ticket adds dedicated read-oriented profile pages so the app stops feeling like a set of disconnected forms.

## Recommended implementation shape
Prefer a dedicated profile page in the main workspace rather than a floating dialog.

Why:
- simpler than adding modal/detail window management
- fits future profile expansion
- easier to reuse for vehicles, places, incidents, gang documentation, etc.
- easier to support back navigation and linked drill-downs

Keep current People / Targets and Organizations pages as the list/management surfaces.
The new profile pages become the drill-down destination.

## Desired UX outcome
### People / Targets list
Each result should show enough information to be useful at a glance:
- display name
- alias/secondary identifier if available
- organization affiliation summary
- maybe linked global person hint if already present and easy to expose

Targets with no affiliation should clearly say:
- `No organization recorded`

### Person profile page
Should include:
- header with primary identity info
- basic details section
- aliases section
- affiliations section
- linked global person section if present
- clean empty states instead of blank boxes
- obvious way to go back

### Organization profile page
Should include:
- header with org name/type/status if available
- aliases section
- memberships section
- child organizations section if present
- clean empty states
- drill-down from member row into person profile

## Technical guidance
### Navigation
Use the simplest existing routed workspace pattern already in the app.
If there is already a navigation service or workspace host, extend it.
If not, add the lightest-weight pattern that supports:
- open person profile
- open organization profile
- back navigation

### Data loading
Use query-driven loading.
Do not load large graphs or keep heavy references alive.
Profile data should be assembled from existing repositories/services.

### UI
Prefer reusable section cards/blocks for:
- profile header
- labeled read-only fields
- empty-state sections
- related entity lists

### People affiliation summary
Keep it simple:
- active/current memberships first
- top 1 to 3 names shown
- overflow text if needed, e.g. `+2 more`

### Empty states
Never leave unlabeled blank regions.
Examples:
- `No aliases recorded`
- `No organization memberships recorded`
- `No child organizations recorded`
