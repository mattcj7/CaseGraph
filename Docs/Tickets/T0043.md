# Ticket: T0043 - Gang documentation supervisor approval workflow v1

## Goal
Add a formal supervisor approval workflow for gang documentation records so documentation can move through controlled review states with durable status history, reviewer attribution, review notes, and defensible auditability.

## Context
CaseGraph now has Gang Documentation module v1, including documentation records, criteria, organization linkage, and profile integration. The next logical step is to make those records reviewable and defensible by adding formal supervisor approval states and status transitions.

Gang documentation must not remain an editable draft forever. It needs an explicit workflow that distinguishes:
- investigator draft work
- records awaiting review
- approved documentation
- inactive documentation
- purge review / purged records

This ticket is about workflow control and review traceability, not packet export or hierarchy visualization.

## Scope
### In scope
- Add formal gang documentation workflow states such as:
  - Draft
  - Pending Supervisor Review
  - Approved
  - Returned for Changes
  - Inactive
  - Purge Review
  - Purged
- Add UI actions to:
  - submit for review
  - approve
  - return for changes
  - mark inactive
  - mark purge review
  - purge
- Add reviewer attribution fields and status history, including at least:
  - reviewer name / reviewer identity reference
  - review timestamp
  - review notes / decision note
  - approval timestamp
  - reason for major state changes
- Add durable workflow/history records for state changes where practical.
- Ensure the documentation UI clearly shows:
  - current state
  - who reviewed it
  - when it was reviewed
  - any review notes / decision notes
- Add guards so inappropriate transitions are blocked.
- Add structured audit/diagnostic logging for workflow transitions.
- Preserve clear separation between:
  - evidence-derived basis
  - analyst-entered judgment
  - supervisor approval state

### Out of scope
- Gang documentation packet export
- Role/permission enforcement beyond the workflow/state layer if full role-aware permissions are not yet in place
- Rivalry/territory modeling
- Organization hierarchy/pyramid visualization
- Automated scoring or approval suggestions
- Broad report builder integration

## Data Model Changes
- Extend gang documentation persistence with workflow/review fields and likely a durable history structure, such as:
  - `GangDocumentationReview`
  - `GangDocumentationStatusHistory`
- Exact names can be refined in implementation, but durable review history is in scope.

## Provenance & Audit Requirements
### Provenance
- Approval states must not alter or blur underlying evidence provenance.
- Gang documentation basis remains distinct from supervisor workflow metadata.
- Existing evidence-linked citations must continue to work unchanged.

### Audit
- Add structured logs and/or audit entries for:
  - submitted for review
  - approved
  - returned for changes
  - marked inactive
  - moved to purge review
  - purged
- Keep behavior consistent with existing CaseGraph audit/diagnostic patterns.
- State transitions must be attributable and reviewable.

## Performance & Resource Management Requirements
- Workflow actions must remain responsive.
- Transition/history queries should be SQLite-safe and bounded.
- Avoid heavy reloads of unrelated profile data for small workflow changes.
- Use safe async and deterministic resource disposal.

## Implementation Notes / Edge Cases
- Keep v1 workflow practical and explicit.
- Returned-for-changes state should not destroy prior review history.
- Purged records may need to remain historically traceable even if hidden from normal views.
- Inactive is not the same as purged.
- UI should clearly label the difference between:
  - documentation content
  - review decision
  - current status
- Do not silently auto-approve or auto-transition records.

## How to Verify Manually
1. Open a target/person profile with gang documentation.
2. Submit the documentation for supervisor review.
3. Confirm the status updates to Pending Supervisor Review.
4. Approve the record and confirm reviewer metadata/history appears.
5. Create another record, submit it, and return it for changes.
6. Confirm the record status, notes, and history update correctly.
7. Mark a record inactive or purge review and confirm the workflow remains clear.
