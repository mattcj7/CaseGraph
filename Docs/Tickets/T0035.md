# Ticket: T0035 - Association graph layout hardening

## Goal
Make Association Graph rendering robust so malformed, degenerate, or edge-case graph data does not crash the rebuild/render path.

## Context
`RebuildAssociationGraph` is currently failing in the MSAGL layout path with a `NullReferenceException` inside `Microsoft.Msagl.Core.Layout.Node.Center` during MDS layout. This indicates the graph render pipeline is passing malformed or unsupported geometry/data into layout, or is not handling degenerate connected components safely. The user should never lose the graph feature because of one malformed node/edge/component; the graph should degrade gracefully, log clearly, and render what it safely can.

## Scope
### In scope
- Audit the Association Graph render/layout pipeline from graph result → MSAGL model → layout → render.
- Validate graph input before layout:
  - non-empty/stable node ids
  - valid node geometry/size/label data
  - edges only referencing existing nodes
- Harden layout behavior for:
  - empty graph
  - single-node graph
  - malformed/disconnected/degenerate components
- Prevent invalid nodes/edges from crashing the whole rebuild command:
  - skip/log invalid items
  - continue rendering valid graph data where possible
- Add a safe fallback path if the preferred MSAGL layout throws:
  - catch/log layout failure
  - retry with a safer/simpler layout or a degraded render path
  - do not crash the overall UI action
- Add structured graph diagnostics:
  - node count
  - edge count
  - invalid node count
  - invalid edge count
  - layout algorithm used
  - fallback used or not
- Add deterministic regression tests for malformed graph input and non-throw behavior.

### Out of scope
- New graph features
- New graph filters or UI redesign
- Changes to graph query semantics / “why linked” logic beyond what is needed for robust rendering
- Switching graph libraries wholesale

## Provenance & Audit Requirements
### Provenance
- Must not weaken existing graph explainability.
- Existing “why linked” details and evidence-backed graph relationships must remain intact for valid graph items.

### Audit
- Preserve existing diagnostics and structured logging patterns.
- This ticket adds graph rebuild/render diagnostics; it does not change evidence/action audit semantics.

## Performance & Resource Management Requirements
- Graph hardening must not introduce long UI blocking work beyond the existing rebuild path.
- Validation should be lightweight and bounded.
- Fallback behavior should prefer graceful degradation over repeated expensive failures.
- Avoid hidden infinite retries.
- Dispose layout/render resources deterministically where applicable.

## Implementation Notes / Edge Cases
- Likely failure sources:
  - node geometry not initialized
  - invalid/duplicate/empty node ids
  - edges referencing missing nodes
  - degenerate connected components fed into MDS layout
- Preferred strategy:
  - validate and normalize graph input first
  - handle empty/single-node components explicitly
  - use MDS only when the component shape is valid/safe
  - if MDS fails, log and fallback to a simpler layout or degraded render
- The UI action should fail gracefully even if some graph content is unusable.
- Keep this ticket focused on graph robustness, not graph feature expansion.
