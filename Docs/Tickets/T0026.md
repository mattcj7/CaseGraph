# Ticket: T0026 - Locations v1 - LocationObservation ingest + Locations page

## Goal
Add **Locations v1** by ingesting geolocation feeds (CSV/JSON/PLIST) into a first-class `LocationObservation` table and exposing a **Locations** page for filtering, review, and provenance drill-down.

## Context
Geolocation is a canonical feature for CaseGraph Offline. We need defensible, provenance-first location analytics where each observation is traceable back to a specific evidence item and locator, and all ingest runs through the job queue with progress/cancellation.

## Scope
### In scope
- Database:
  - Add `LocationObservation` table + indexes (migration-first).
- Ingest:
  - Add `LocationsIngest` job that parses supported location artifacts and writes `LocationObservation` rows with provenance.
  - Supported inputs (v1):
    - CSV: columns for timestamp + lat + lon (+ optional accuracy/altitude/speed/heading)
    - JSON: array/object forms containing lat/lon/time (+ optional accuracy/etc.)
    - PLIST: best-effort parse for location arrays/dicts with lat/lon/time (+ optional accuracy/etc.)
  - Schema-tolerant parsing:
    - unknown fields skipped with structured logs
    - single bad row/file does not crash full ingest
  - Precise SourceLocator per observation (file + row index / json pointer / plist key-path).
- UI:
  - Add **Locations** page in left-nav.
  - Locations table with:
    - timestamp (local display)
    - latitude/longitude
    - accuracy (if present)
    - source label/type
    - linked subject (Target/Global Person) if resolvable/available
    - actions: **View Source**, **Copy Citation**
  - Filters (v1):
    - date range (from/to)
    - subject filter (optional, reuse existing picker if available)
    - min/max accuracy (optional)
    - source type (CSV/JSON/PLIST)
  - Paging/virtualization for large datasets.
- Audit:
  - Log:
    - `LocationsIngestStarted/Completed` (or single `LocationsIngestCompleted`) with case id, evidence ids, counts, correlation id
    - `LocationsSearchExecuted` when user runs location queries/filters

### Out of scope
- Offline map rendering / tile packs / heatmaps (future ticket)
- Geofencing / clustering / route reconstruction (future)
- Automatic subject attribution/merges (no silent merges)
- Cross-case location aggregation (future)

## Data Model Changes
- Entity/Table: `LocationObservation`
  - LocationObservationId: INTEGER (pk)
  - CaseId: TEXT/INTEGER (fk)
  - ObservedUtc: TEXT (ISO 8601 UTC)
  - Latitude: REAL
  - Longitude: REAL
  - AccuracyMeters: REAL NULL
  - AltitudeMeters: REAL NULL
  - SpeedMps: REAL NULL
  - HeadingDegrees: REAL NULL
  - SourceType: TEXT (CSV|JSON|PLIST|Other)
  - SourceLabel: TEXT NULL (filename/feed name)
  - SubjectType: TEXT NULL (Target|GlobalPerson)
  - SubjectId: TEXT/INTEGER NULL
  - SourceEvidenceItemId: TEXT/INTEGER
  - SourceLocator: TEXT
  - IngestModuleVersion: TEXT
  - CreatedUtc: TEXT

Indexes (expected):
- (CaseId, ObservedUtc DESC)
- (CaseId, Latitude, Longitude) optional
- (CaseId, SubjectType, SubjectId, ObservedUtc DESC) if subject filter is common

## Provenance & Audit Requirements
### Provenance
All derived `LocationObservation` rows must store:
- SourceEvidenceItemId
- SourceLocator (precise: CSV row number, JSON pointer, PLIST key-path)
- IngestModuleVersion

### Audit
Log:
- Locations ingest start/end (counts, evidence ids, correlation id)
- Locations searches (filters, result count, correlation id)

## Performance & Resource Management Requirements
- Use async/await for I/O; never block UI thread.
- Stream parse large files where possible.
- Dispose IDisposable resources deterministically.
- Support CancellationToken for ingest and queries.
- Parser stability: unknown fields skipped; one bad artifact must not crash full ingest.
- Paging/virtualization: do not load all observations into memory.

## Implementation Notes / Edge Cases
- Timestamp parsing:
  - accept multiple common formats; normalize to UTC for storage
  - if timestamp missing/invalid: skip row with structured warning
- Coordinate validation:
  - lat in [-90, 90], lon in [-180, 180]; invalid rows skipped and logged
- PLIST:
  - best-effort extraction; log summary of matched keys
- Provenance locators:
  - CSV: `csv://<file>#row=<n>`
  - JSON: `json://<file>#ptr=<json-pointer>`
  - PLIST: `plist://<file>#keyPath=<path>`
