# Ticket: T0040 - Entity Profile Viewer Navigation for People and Organizations

## Goal
Create dedicated profile viewers for People / Targets and Organizations so investigators can double-click into a readable entity profile, see affiliations and memberships at a glance, and navigate between linked entities without relying on hover-only labels.

## Context
The current skeleton UI is proving out the underlying entity and organization system, but the workflow still feels form-first instead of investigation-first. A person or organization should behave like a first-class case object with a dedicated profile surface. This ticket improves usability and discoverability without expanding into cross-case registry, gang documentation workflows, or heavy analytics.

## Scope
### In scope
- Add a dedicated Person / Target Profile page in the main workspace.
- Add a dedicated Organization Profile page in the main workspace.
- Support double-click navigation from:
  - People / Targets list results
  - Organizations list results
  - Organization membership rows to person profile
  - Person affiliation rows/chips to organization profile
- Show organization affiliations directly in the People / Targets results list.
- Replace hover-only field discovery with persistent labels / section headers on current People / Targets and Organizations screens where blank fields are visible.
- Add clear empty states for sections with no data.
- Add basic back / return navigation so users can return to their originating workspace.
- Keep existing create/save flows working while adding profile-view navigation.
- Add an explicit Open Profile action in addition to double-click where practical.

### Out of scope
- Full cross-case Global Target Registry implementation.
- Gang Documentation workflow.
- New fuzzy matching or merge logic.
- New report/export templates.
- New graph analytics.
- Large schema refactor of targets, organizations, or memberships.
- Full shell redesign or detachable window framework.

## Acceptance Criteria (Testable)
- [ ] Double-clicking a person/target in the People / Targets results opens that entity’s dedicated profile page.
- [ ] Double-clicking an organization in the Organizations results opens that entity’s dedicated profile page.
- [ ] Double-clicking a membership row on an organization profile opens the linked person profile.
- [ ] Double-clicking an affiliation on a person profile opens the linked organization profile.
- [ ] The People / Targets results list shows organization affiliation summary text for targets with memberships.
- [ ] Targets with no recorded affiliations display a clear fallback such as `No organization recorded`.
- [ ] Person profile pages display a readable header plus sections for overview/basic details, aliases, affiliations, and linked global person info if present.
- [ ] Organization profile pages display a readable header plus sections for overview/basic details, aliases, child organizations if present, and memberships.
- [ ] Current People / Targets and Organizations screens no longer show visible blank fields that depend only on tooltip hover text to explain their purpose.
- [ ] Profile pages and updated list items use persistent labels/headers and clear empty-state text.
- [ ] Opening a profile does not freeze the UI and returns control within normal interactive expectations for seeded local data.
- [ ] Existing save/edit behavior for targets, organizations, aliases, and memberships continues to work after this change.
- [ ] No silent merges or new automatic link approvals are introduced by this ticket.

## Provenance & Audit Requirements
### Provenance
All derived records created must store:
- SourceEvidenceItemId
- SourceLocator (precise: XML node id / plist key-path / Excel sheet+row / HTML file path + stable locator)
- IngestModuleVersion

Notes for this ticket:
- This is primarily a UI/routing ticket and should reuse existing records rather than create new persistent derived data.
- If any persisted read-model/cache/projection is introduced for affiliation summaries or profile loading, it must preserve citation linkage back to the underlying membership/evidence or analyst-entered record.
- Person affiliation displays and organization membership displays must remain explainable and should never imply unsupported provenance.

### Audit
Log these actions/events:
- Existing create/update/delete actions for targets, aliases, organizations, and memberships must continue to be audited.
- Existing link/unlink actions between target and global person must continue to be audited.
- No new audit entry is required for passive navigation such as opening a profile page or returning back.
- If implementation adds a new explicit Open Profile command, do not audit it unless product-wide navigation auditing is later adopted.

## Performance & Resource Management Requirements
- Use async/await for I/O; never block UI thread.
- Use streaming for large files; avoid loading entire evidence blobs into memory when possible.
- Dispose all IDisposable resources deterministically (streams/zip/db contexts/file handles).
- Avoid WPF event-handler leaks (unsubscribe or use WeakEventManager where appropriate).
- Support CancellationToken for long-running operations (imports/indexing/search).
- Parser stability: unknown fields skipped with structured logs; one bad artifact must not crash full ingest.
- Imports must not freeze UI; memory usage must not grow unbounded during long imports.
- Profile loading and affiliation/member lookups must be query-driven and not load unrelated entity graphs into memory.
- Repeated profile navigation should reuse lightweight view models or cached query results carefully without holding large object graphs indefinitely.
- Double-click/open-profile navigation must stay responsive on seeded cases with multiple memberships and aliases.

## Implementation Notes / Edge Cases
- Prefer a dedicated page in the main content workspace over a floating dialog for MVP.
- Keep the current People / Targets and Organizations pages as list/management surfaces; the new profile pages are the drill-down destination.
- Add a simple Open Profile button/context action in addition to double-click for discoverability and accessibility.
- If the People / Targets list is currently a ListBox rather than a grid, update the item template to show:
  - display name
  - primary alias
  - affiliation summary
  rather than forcing a full grid rewrite in this ticket.
- Person affiliation summary should prefer active/current memberships first; if multiple exist, show the top 1–3 plus overflow text like `+2 more`.
- Organization profile should show membership rows with role, status, confidence, and last-confirmed date when available.
- Empty sections should never appear as unlabeled blank rectangles. Use labels and text like:
  - No aliases recorded
  - No organization memberships recorded
  - No child organizations recorded
- If an entity is deleted or becomes unavailable between selection and navigation, show a non-crashing `Record no longer available` state.
- Preserve provenance language:
  - analyst-entered relationships should remain clearly labeled
  - evidence-derived relationships should remain clearly labeled
- Avoid introducing new merge behavior through profile navigation.
- Keep navigation predictable:
  - back button returns to prior workspace
  - reopening the same profile should refresh/focus rather than stacking duplicates if that is easy to implement
- Membership rows and affiliation chips must tolerate partial data:
  - missing role
  - missing confidence
  - missing last-confirmed date
  - inactive membership states

## How to Verify Manually
1. Open a workspace with at least one target and one organization membership.
2. Go to People / Targets and confirm the list now shows affiliation summary text for affiliated targets.
3. Double-click a target and confirm a dedicated person profile page opens.
4. Confirm the person profile has persistent labels/section headers and no unlabeled blank regions.
5. From the person profile, open an affiliated organization and confirm it navigates to the organization profile.
6. Confirm the organization profile shows aliases, memberships, and empty-state text for any missing sections.
7. Double-click a member row in the organization profile and confirm it opens that person’s profile.
8. Return back to the originating page and confirm navigation context is still usable.
9. Edit/save an existing target or organization using the current management flow and confirm behavior still works.
10. Review audit history and confirm save/edit/link operations are still logged, while passive profile navigation is not.
11. Test with a target that has no affiliations and confirm the UI shows a clear fallback instead of an empty/unlabeled area.
12. Test with partial membership data and confirm the profile remains readable and stable.
