# Ticket: T0030 - Case-open lazy initialization and feature-scoped readiness

## Goal
Make CaseGraph feel fast and predictable by moving heavy readiness work out of global app startup and into:
- case-open time when the user opens a case
- feature-open time when a specific feature actually needs preparation

The app shell should open quickly, and expensive maintenance should be visible, bounded, and scoped to the case or feature that needs it.

## Context
Recent startup work showed that CaseGraph can spend several minutes doing database/index maintenance before the main window becomes usable. As evidence volume grows, that approach will only get worse. The design requires no UI blocking during long operations, guided workflows, and progress visibility for expensive work. This ticket shifts the architecture toward:
- fast shell startup
- case-scoped readiness
- feature-scoped readiness
- background/deferred maintenance where safe

## Scope
### In scope
- Define and implement a two-stage startup model:
  1) **App startup** (minimal, global, fast)
  2) **Case open / feature open** (scoped readiness checks and deferred maintenance)
- Reduce global startup work to only what is required to safely show the shell:
  - logging bootstrap
  - workspace path/open checks
  - required migration safety checks
  - fatal/corruption checks
- Move non-critical readiness work out of global startup when safe:
  - message FTS ensure / backfill
  - feature-specific cache/index warmups
  - other case-specific maintenance that does not need to block shell launch
- Add a **case-open readiness flow** with visible progress for work that must happen when a case is opened.
- Add **feature-scoped readiness checks** for expensive features such as:
  - Search
  - Timeline
  - Reports
  - Incident Window
- If a feature needs background preparation, show clear status instead of freezing the whole app.
- Preserve correctness, auditability, provenance, and existing diagnostics.

### Out of scope
- New evidence parsers
- New product features unrelated to startup/case-open/feature-open readiness
- Visual redesign of the shell
- Window chrome controls (min/max/close) — separate ticket
- Broad refactor of unrelated modules

## Acceptance Criteria (Testable)
- [ ] Main app shell opens quickly without waiting for non-critical case/feature maintenance.
- [ ] Required global startup checks still run safely and do not compromise correctness.
- [ ] Opening a case runs visible case-scoped readiness work when needed.
- [ ] Search/Timeline/Reports/Incident Window do not require unrelated heavy readiness work before the shell is shown.
- [ ] Feature-specific maintenance is triggered only when required by that feature, or deferred safely in background.
- [ ] Long-running readiness tasks are visible and do not appear as silent hangs.
- [ ] Existing structured logs clearly show whether work occurred at:
  - [ ] app startup
  - [ ] case open
  - [ ] feature open
- [ ] Deterministic tests cover the new readiness split and prevent regression to heavy global startup.
- [ ] `dotnet test` passes.

### Audit

  - feature name if applicable

## Performance & Resource Management Requirements
- App startup must remain thin and avoid long blocking maintenance.
- Heavy readiness work must be visible, bounded, and cancellable where safe.
- Do not load all case data or rebuild all feature state at shell launch.
- Prefer lazy initialization and background work over eager global warmups.
- Dispose resources deterministically.
- Avoid UI thread blocking, deadlocks, and hidden loops/retries.

## Implementation Notes / Edge Cases
- Separate readiness into three buckets:
  1) **Must-run before shell**
  2) **Must-run before case opens**
  3) **Must-run before feature opens**
- Message FTS/search readiness is a likely candidate to move out of global startup.
- If feature readiness is not complete:
  - the feature should show a clear loading/preparing state
  - unrelated features should remain usable
- If a case has no relevant data for a feature, readiness should no-op quickly.
- Keep the user experience simple:
  - shell opens fast
  - open case shows progress if needed
  - feature open shows status if needed
- Prefer explicit readiness contracts over scattered ad hoc checks.

## Test Plan
### Unit tests
- readiness phase classification behaves as expected
- startup orchestration no longer triggers feature/case-only maintenance globally
- case-open readiness requests expected tasks
- feature-open readiness requests only its own dependencies

### Integration tests (if applicable)
- shell startup completes without triggering non-critical FTS backfill on an already-initialized workspace
- opening a case triggers case-scoped readiness instead of global startup work
- opening Search/Timeline/Reports/Incident Window triggers only the minimum required feature readiness
- logs identify readiness phase clearly

## How to Verify Manually
1. Launch the app on a real workspace and confirm the shell appears quickly.
2. Confirm the app does not sit hidden in Task Manager doing long silent startup work.
3. Open a case and confirm any required case readiness work is visible.
4. Open Search, Timeline, Reports, and Incident Window and confirm each feature becomes ready without blocking unrelated parts of the app.
5. Review logs and confirm readiness work is labeled as startup, case-open, or feature-open.
