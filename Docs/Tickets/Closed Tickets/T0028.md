# Ticket: T0028 - Startup and page activation stabilization sweep

## Goal
Stabilize app loading by auditing and fixing the startup pipeline and page activation paths most likely causing current load failures, while adding regression coverage and clearer diagnostics so future hangs/crashes are easy to pinpoint.

## Context
Recent behavior suggests multiple stability regressions:
- one run crashed during `ReportsViewModel` construction/init-order
- another run stopped after migrations completed, before schema verification/host startup finished
- Reports and Timeline tabs have also failed to open, while Association Graph appeared functional

The uploaded logs indicate the problem is concentrated in the startup lifecycle and page activation flow, not a general app-shell failure. This should be treated as a focused stabilization ticket, not a whole-repo refactor.

## Scope
### In scope
- Audit and stabilize the startup lifecycle from:
  - host build
  - workspace initialization
  - migration completion
  - post-migration maintenance steps (including any FTS ensure/backfill steps)
  - schema verification
  - finalize/start host
  - main window show
- Audit and stabilize page activation / navigation for:
  - Reports
  - Timeline
  - Incident Window (if it participates in eager resolution or recent navigation changes)
- Fix constructor/init-order bugs in eagerly constructed viewmodels
- Verify DI registrations for startup-critical services and major pages/viewmodels
- Add more granular structured logging around startup and page-activation boundaries
- Ensure page activation failures are logged clearly instead of appearing as silent no-ops
- Add deterministic regression tests for:
  - startup progression past migration completion
  - safe construction/resolution of Reports and Timeline viewmodels
  - page activation/navigation tests if a pattern already exists

### Out of scope
- New product features
- Broad architectural refactor of unrelated modules
- Visual redesign
- Performance optimization outside what is needed to prevent load failures
- Refactoring currently stable pages with no evidence of regression

## Acceptance Criteria (Testable)
- [ ] App successfully loads to main window on the previously failing workspace.

## Implementation Notes / Edge Cases
- Treat this as a stabilization sweep, not a whole-repo refactor.
- Likely hotspots:
  - startup pipeline between migration completion and schema verification/finalize
  - post-migration maintenance steps such as ensuring message FTS objects
  - eager DI/viewmodel construction
  - observable-property change handlers firing before state initialization
  - route registration / page activation exceptions
- For MVVM Toolkit observable properties:
  - avoid firing side-effect handlers during incomplete construction
  - initialize backing state first
  - use an initialization guard where needed
- Add explicit logs around:
