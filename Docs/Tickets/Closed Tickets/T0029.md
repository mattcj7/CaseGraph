# Ticket: T0029 - Startup progress window and deferred startup work

## Goal
Add a visible **startup progress window** so long app startup no longer looks like a silent hang, and defer any non-critical startup work until after the main window is shown when safe to do so.

## Context
Recent stabilization work showed that startup can take several minutes on real workspaces, especially around post-migration maintenance like message FTS ensure/backfill. Even when startup eventually succeeds, the current experience feels broken because the app appears only in Task Manager with no visible UI for too long. We need startup to be transparent, guided, and trustworthy for non-technical users.

## Scope
### In scope
- Add a small **startup progress window** that appears immediately on app launch.
- Show real startup stages, not a fake spinner.
- Stages should include, at minimum:
  - Building host
  - Opening workspace
  - Loading/applying migrations
  - Ensuring message search index / FTS objects
  - Verifying schema
  - Finalizing startup
  - Opening main window
- Show:
  - current stage label
  - optional detail text / sub-status
  - elapsed time
- Wire startup-stage reporting from the real startup pipeline to the progress window.
- Close the startup progress window automatically once the main window is ready.
- If safe, defer non-critical startup work until after the main window is shown.
- If startup fails:
  - show a clear startup failure message
  - keep diagnostics path discoverable (logs folder / dump folder if already supported)
- Preserve existing structured startup logging and correlation ids.

### Out of scope
- Visual redesign of the whole shell
- Background tray mode
- Fancy animations or branding polish
- Window chrome controls (min/max/close buttons) beyond what the startup window needs
- Refactoring unrelated startup components

## Acceptance Criteria (Testable)
- [ ] A startup progress window appears quickly on launch, before long-running startup work completes.
- [ ] The progress window shows real startup stages and updates as startup advances.
- [ ] On a slow real workspace, the user can see which stage is currently running.
- [ ] When startup completes successfully, the main window opens and the startup progress window closes.
- [ ] When startup fails, the user sees a clear failure state instead of a silent hang.
- [ ] Non-critical startup work is deferred until after the main window if safe and documented.
- [ ] Existing structured logs still show startup stages/correlation ids.
- [ ] Deterministic tests cover startup stage reporting / progress window orchestration where feasible.
- [ ] `dotnet test` passes.

## Performance & Resource Management Requirements
- The startup progress window must appear quickly and not itself block startup.
- Do not add unnecessary startup work.
- Prefer deferring non-critical maintenance until after the main window is visible.
- Keep startup stage reporting lightweight and thread-safe.
- Dispose startup resources deterministically.
- Avoid deadlocks and cross-thread UI access bugs.

## Implementation Notes / Edge Cases
- Progress should be stage-based, not fake percentage-based, unless percentage is truly available.
- The startup window should tolerate unknown-duration stages.
- If a stage is retried or resumed, surface useful text instead of freezing the displayed stage forever.
- If deferring FTS or similar work, ensure search remains safe:
  - either fully ready
  - or clearly marked/rebuilt in the background without corrupting state
- If startup fails before main window creation, the startup progress window should remain visible long enough to show the failure state.
- Reuse existing startup diagnostics/logging instead of inventing a second diagnostics system.

## How to Verify Manually
1. Launch the app on the real workspace that previously took several minutes to open.
2. Confirm a startup progress window appears quickly.
3. Confirm stage text changes as startup advances.
4. Confirm the main window eventually opens and the startup progress window closes.
5. If possible, force a startup failure and confirm the failure state is visible.
6. Review the latest app log and confirm startup stages still have clear structured entries.
