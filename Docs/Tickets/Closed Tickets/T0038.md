# Ticket: T0038 - Open incident workspace v1

## Goal
Create a first-class Open Incident Workspace so investigators can enter an unsolved incident, define the offense window and scene area, and cross-reference that incident against messages, locations, timeline events, and saved findings.

## Context
CaseGraph now has enough foundation to support a true investigative workflow: startup/readiness is improved, search is usable, graphing exists, locations can parse, and synthetic evidence generation exists. The next logical step is to let investigators work from an incident outward instead of only from imported evidence inward.

An open incident record should serve as a durable investigative anchor for things like:
- unsolved drive-by shootings
- shots fired
- assaults
- homicides
- retaliatory incidents

The workspace should let the investigator define the incident and then run time/location-based cross-reference queries across existing CaseGraph data.

## Scope
### In scope
- Add a first-class **Open Incident Workspace** feature.
- Add an incident record model with at least:
  - incident id
  - title
  - incident type
  - status
  - summary/notes
  - primary occurrence time
  - offense window start/end
- Add support for one or more incident locations with:
  - label
  - latitude
  - longitude
  - radius meters
  - notes
- Add a UI workflow to:
  - create an incident
  - edit an incident
  - save an incident
  - select an incident and run cross-reference
- Cross-reference v1 should query:
  - messages in the offense window
  - locations within the offense window and incident radius
- Results should be shown in a dedicated incident results view with at least:
  - message results
  - location results
  - pinned/saved findings section
- Add timeline integration:
  - the incident appears as a timeline anchor/marker
  - related messages and location hits can be shown around the incident window
- Allow investigators to pin/save relevant results back to the incident workspace.
- Preserve provenance and diagnostics.
- Add structured logging for incident create/update/run flows.

### Out of scope
- Full scoring/ranking engine
- Gang organization linkage in this ticket
- Rivalry/territory modeling
- Incident-to-case automation beyond simple linking if already available
- Advanced map rendering
- Multi-incident comparison workflows
- Broad narrative/report generation

## Data Model Changes
- Add incident-focused data structures such as:
  - `IncidentRecord`
  - `IncidentLocation`
  - optional `IncidentPinnedResult` or equivalent saved-finding structure
- Exact table/entity names can be refined in implementation, but incident persistence is in scope.

## Provenance & Audit Requirements
### Provenance
- Message and location results returned through the incident workspace must preserve existing provenance behavior:
  - SourceEvidenceItemId
  - SourceLocator
  - existing View Source / Copy Citation behavior where applicable

### Audit
- Preserve existing diagnostics and logging patterns.
- Add structured logs for:
  - incident created
  - incident updated
  - incident cross-reference run
  - result counts returned

## Performance & Resource Management Requirements
- Cross-reference queries must not freeze the UI.
- Use existing readiness/background patterns where applicable.
- Keep v1 bounded:
  - offense window filtering first
  - radius filtering on incident locations
- Avoid unnecessary full-dataset scans where existing indexes/services can be reused.
- Dispose resources deterministically.

## Implementation Notes / Edge Cases
- Keep v1 focused on a single incident workspace, not a full incident intelligence platform.
- Support multiple locations per incident, but keep the UI simple.
- If exact incident time is unknown, investigator should still be able to define an offense window manually.
- If no results are found, the workspace should still show the saved incident and a clean empty state.
- Timeline integration should anchor the incident without breaking existing timeline behavior.
- Prefer reuse of existing Timeline, Search, and Location query infrastructure where it fits cleanly.
- Pinned/saved findings can be lightweight in v1 as long as they are durable and tied to the incident.

## How to Verify Manually
1. Open CaseGraph and create a new incident in the Open Incident Workspace.
2. Enter a title, incident type, offense window, and at least one scene location with radius.
3. Save the incident.
4. Run cross-reference.
5. Confirm messages and locations inside the window/radius are returned.
6. Confirm the incident appears as a timeline anchor/marker.
7. Pin at least one result and confirm it remains associated with the incident.
