# Ticket: T0027 - Incident Window v1 - offense window comms + geo hits

## Goal
Implement an **Incident Window** workspace (wizard + results view) that lets an investigator select:
- a time window (e.g., -6h to +12h around offense time)
- an optional scene center (lat/lon) + radius

…and returns a prioritized, explainable set of:
- message/timeline hits (comms)
- location hits (within radius/time)
- optional “co-location candidates” (two subjects near the scene in the window)

All results must support **View Source** and **Copy Citation** and the action must be **audited**.

## Context
The design calls for an **Incident Window Wizard** that selects a time window (and optional radius) and returns prioritized comms/geo hits for ready-to-export timeline slices. :contentReference[oaicite:2]{index=2}  
Strategic planning marks the Incident Window workspace as a high-value module because it makes offense-date comms, proximity, and co-location analysis fast and explainable. :contentReference[oaicite:3]{index=3}

## Scope
### In scope
- Add **Incident Window** entry in navigation (or place it under Timeline/Locations as “Incident Window”).
- Wizard inputs (v1):
  - StartUtc/EndUtc (or preset offsets: -6h/+12h buttons)
  - Optional center point:
    - manual lat/lon entry (required only if radius enabled)
    - radius meters (e.g., 50–5000m)
  - Optional subject filter (Target/Global Person) to focus the results
- Results view (v1) with three tabs/panels:
  1) **Comms hits** (MessageEvents) within the time window
     - newest first
     - shows participants (resolved via existing resolution logic)
     - snippet (bounded) + citations
  2) **Geo hits** (LocationObservation) within time window and radius (if provided)
     - newest first
     - shows subject (if available), lat/lon, accuracy, citations
  3) **Co-location candidates** (best-effort)
     - pairs of subjects with observations within X meters within Y minutes
     - explain “why” by citing the underlying LocationObservation ids/locators
- Row actions everywhere:
  - **View Source**
  - **Copy Citation**
  - Open related item (open message thread, open subject, open location detail) if already supported
- Audit logging:
  - `IncidentWindowExecuted` with parameters and counts
  - `IncidentWindowExported` if export is implemented in this ticket
- Optional export (v1, keep lightweight):
  - “Export Incident Window Packet” using existing report infrastructure to write a small PDF/HTML packet containing:
    - parameters (window + radius)
    - selected comms + geo tables
    - citations list

### Out of scope
- Full map rendering / offline tile packs / heatmaps (future)
- Clustering/frequent places/route reconstruction (future)
- Automatic subject attribution or merges (no silent merges)
- Cross-case incident windows (future)

## Provenance & Audit Requirements
### Provenance
All displayed results must support provenance drill-down:
- Message hits must cite underlying evidence (SourceEvidenceItemId + SourceLocator).
- Geo hits must cite underlying evidence (SourceEvidenceItemId + SourceLocator).
- Co-location candidates must cite the underlying LocationObservation rows used to compute the pair.

### Audit
Log:
- `IncidentWindowExecuted` (case id, window start/end, radius enabled + center/radius, optional subject filter, result counts, correlation id)
- If export shipped: `IncidentWindowExported` (output path + parameters)

## Performance & Resource Management Requirements
- Use async/await; never block UI thread.
- Paging/virtualization for result lists (do not load everything into memory).
- Radius query approach:
  - Apply time filter in SQL first.
  - Apply bounding-box prefilter in SQL when radius is enabled.
  - Apply precise Haversine in-app over the reduced candidate set.
- Dispose DB contexts/readers deterministically.
- All operations accept CancellationToken.
- Fail-soft: a single bad row must not crash the query; log and continue.

## Implementation Notes / Edge Cases
- If radius is enabled but center is missing → show validation error.
- If no LocationObservation exists yet → show comms-only with clear empty state.
- Timestamp handling:
  - store/query in UTC, display in local.
- Co-location (best-effort):
  - keep thresholds configurable defaults (e.g., 100m within 10 minutes)
  - show “why” with citations to the two observations (and times/distances)
- Keep this ticket small:
  - prioritize correctness + explainability over fancy ranking.

2. Run Incident Window with a known time window; confirm comms results.
