# Ticket: T0032 - Performance instrumentation and budgets

## Goal
Add clear, structured performance instrumentation and budget thresholds so CaseGraph can detect, log, and surface slow paths before they turn into user-visible hangs.

## Context
Recent reinforcement work improved startup and Search responsiveness, but the pattern is clear: expensive work keeps moving between startup, case-open, and feature-open flows. We need a consistent way to measure, log, and enforce performance expectations across the app so optimization is driven by evidence instead of guesswork.

## Scope
### In scope
- Add a reusable performance timing/instrumentation layer for key user flows.
- Instrument, at minimum:
  - app startup
  - case open
  - Search readiness
  - Search query execution
  - Timeline readiness/open
  - Reports readiness/open
  - Incident Window readiness/query
  - import/index maintenance jobs where feasible
- Define budget thresholds for:
  - startup
  - case open
  - feature readiness
  - feature query execution
- Emit structured logs for:
  - operation start
  - operation completion
  - elapsed milliseconds
  - slow-path warning when a budget is exceeded
  - correlation id
  - case id / feature name where applicable
- Keep instrumentation lightweight and reusable.
- Add tests for timing/logging behavior where practical.
- Update diagnostics so slow-path events are easy to identify in logs.

### Out of scope
- New performance optimizations unrelated to instrumentation
- UI redesign
- Window chrome controls
- Full telemetry/analytics platform
- External services or network reporting

## Acceptance Criteria (Testable)
- [ ] Key flows listed in scope emit structured start/completion timing logs.
- [ ] Budget thresholds are defined in code/config for startup, case-open, and major feature readiness/query flows.
- [ ] When a threshold is exceeded, a structured slow-path warning is logged.
- [ ] Instrumentation is lightweight and does not materially slow the app.
- [ ] Logs clearly distinguish startup, case-open, feature-open, and query execution timing.
- [ ] Deterministic tests cover:
  - [ ] elapsed time capture / reporting
  - [ ] slow-path warning emission
  - [ ] correct operation/feature labeling
- [ ] `dotnet test` passes.

## Provenance & Audit Requirements
### Provenance
- N/A for evidence-derived records.
- Must not weaken existing provenance behavior in instrumented flows.

### Audit
- Preserve existing structured diagnostics and correlation ids.
- Performance instrumentation logs must remain local/offline and consistent with current logging patterns.
- This ticket does not add user action audit records; it adds diagnostic performance logs.

## Performance & Resource Management Requirements
- Instrumentation must be lightweight.
- Avoid high-allocation or chatty logging in tight loops.
- Prefer scoped timing around major operations, not per-row/per-record tracing.
- Do not block the UI thread to gather timing.
- Dispose timers/scopes deterministically if needed.
- Slow-path thresholds should be easy to adjust without broad refactors.

## Implementation Notes / Edge Cases
- Use a reusable pattern so future tickets can instrument new flows consistently.
- Suggested timing labels:
  - `Startup`
  - `CaseOpen`
  - `FeatureReadiness`
  - `FeatureQuery`
  - `ImportMaintenance`
- Suggested dimensions:
  - feature name
  - case id
  - operation name
  - elapsed ms
  - threshold ms
  - slow-path flag
- Prefer structured logging over string-only messages.
- Do not spam logs for tiny operations.
- Keep it simple and practical: enough to find real bottlenecks quickly.
