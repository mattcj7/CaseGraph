# Ticket: T0034 - Background maintenance manager and readiness banners

## Goal
Centralize deferred maintenance into a visible, reusable background maintenance manager so expensive readiness/repair work no longer blocks investigators and feature pages can show clear “preparing” or “maintenance in progress” states.

## Context
Recent reinforcement work moved heavy readiness out of startup and reduced Search blocking, but the same pattern can still appear in other features over time. CaseGraph needs a consistent way to:
- queue background maintenance safely
- track status centrally
- surface visible readiness banners/messages in feature pages
- avoid hidden hangs and repeated ad hoc maintenance logic

The design requires no UI blocking during long operations and emphasizes guided, predictable workflows with progress visibility. This ticket formalizes that pattern into a reusable maintenance system.

## Scope
### In scope
- Add a reusable **background maintenance manager** that can:
  - register named maintenance tasks
  - deduplicate overlapping requests
  - track status (Idle / Pending / Running / Completed / Failed)
  - expose progress/status text where available
  - support cancellation where safe
- Define maintenance task categories for current needs such as:
  - message FTS ensure/rebuild
  - search preparation
  - future timeline/report/index preparation
- Add **feature readiness banners/status messages** so pages can show:
  - preparing
  - maintenance in progress
  - completed
  - failed (with clear next step where appropriate)
- Integrate the manager into at least:
  - Search
  - Timeline
  - Reports
  - Incident Window if affected by deferred preparation
- Ensure maintenance tasks run in the background or job-driven flow where safe, without freezing the app.
- Add structured logs for:
  - maintenance request queued
  - maintenance started
  - maintenance completed
  - maintenance failed
  - maintenance skipped/deduped
- Preserve existing provenance, diagnostics, and performance instrumentation.

### Out of scope
- New investigative features
- Full diagnostics UI redesign
- Major visual redesign of feature pages
- Replacing the existing JobQueue wholesale
- Broad refactor of unrelated services

## Provenance & Audit Requirements
### Provenance
- Must not weaken existing provenance behavior.
- Search/results/timeline/report actions must continue to support View Source and Copy Citation unaffected by maintenance state changes.

### Audit
- Preserve existing structured diagnostics and correlation ids.
- This ticket adds diagnostic/logging behavior for maintenance lifecycle; it does not replace existing AuditLog semantics for user actions.

## Performance & Resource Management Requirements
- Maintenance must not block the UI thread.
- Prefer background execution and bounded work.
- Support cancellation where safe.
- Deduplicate redundant maintenance requests to avoid repeated expensive work.
- Avoid hidden loops and repeated rebuilds.
- Dispose resources deterministically.

## Implementation Notes / Edge Cases
- Keep the manager small and practical:
  - task key/name
  - current state
  - last run time
  - optional progress text
  - error message if failed
- A feature page should be able to ask:
  - is my readiness current?
  - if not, what maintenance is running?
  - what should I show the user right now?
- Prefer a reusable banner/status component over custom one-off states per page.
- If maintenance fails:
  - show a clear state
  - do not freeze the feature page
  - allow retry where appropriate
- If an operation is already running, a second request should attach to the same task instead of starting a duplicate run.
- Reuse the existing job/readiness/performance instrumentation patterns where helpful instead of inventing a second system.

## Test Plan
### Unit tests
- manager starts and tracks maintenance tasks correctly
- duplicate requests are deduplicated/coalesced
- failed tasks transition to failed state with message
- banner/viewmodel state maps correctly from maintenance state

### Integration tests (if applicable)
- Search request triggers shared maintenance and shows preparing state without freezing
- second request during active maintenance does not start duplicate work
- completion transitions feature back to ready state

## How to Verify Manually
1. Launch the app and open a case.
2. Open Search on a workspace that requires maintenance and confirm a visible preparing/maintenance state appears.
3. Confirm the app remains responsive while maintenance runs.
4. Open another affected feature and confirm it shows status instead of silently hanging.
5. Trigger the same maintenance request twice and confirm duplicate work is not started.
6. Review logs and confirm maintenance lifecycle events are recorded clearly.
