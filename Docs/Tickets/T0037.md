# Ticket: T0037 - Message parser adaptability and canonical schema mapping

## Goal
Make message ingest more adaptable by introducing a canonical internal message schema, header/field alias mapping, and detector-driven compatibility logic so CaseGraph can parse a broader range of message export shapes without requiring exact vendor-perfect templates.

## Context
The synthetic evidence generator in T0036 successfully created uploadable datasets, and locations parsed correctly, but message ingest reported: “no message parser is available for this evidence.” That result exposed a larger product need: CaseGraph’s message parsing must be more adaptable to variable export shapes because we will not have every real Cellebrite dump, spreadsheet, social media return, or warrant export format available up front.

Instead of hardcoding rigid format expectations, CaseGraph should normalize recognized data into a standard internal message model and support broader compatibility through:
- detector-driven parser routing
- header alias matching
- required vs optional field handling
- parse diagnostics and warnings

## Scope
### In scope
- Define a canonical internal message schema for normalized ingest output, including fields such as:
  - message content/body
  - sender
  - receiver(s)
  - timestamp/date-time
  - platform
  - thread/chat id
  - direction
  - attachment count or attachment presence
  - source file / source locator
  - parser family / parser version
  - parse warnings / confidence indicators where appropriate
- Add adaptable header/field alias matching for CSV/XLSX-like message exports.
- Improve detector/routing logic so message parsers can recognize broader message-export shapes by:
  - file type
  - header aliases
  - workbook/sheet structure if applicable
  - basic shape heuristics
- Distinguish required vs optional fields:
  - parse what is safe when optional fields are missing
  - log warnings instead of hard-failing where possible
- Add parser diagnostics/logging such as:
  - parser family selected
  - fields matched to canonical schema
  - unmapped columns
  - skipped/unknown fields
  - row counts parsed/skipped
- Update the synthetic evidence generator compatibility if needed so at least one generated profile can successfully flow through the real message ingest path using the adaptable parser logic.
- Add deterministic regression tests for schema mapping and compatibility behavior.

### Out of scope
- Full parser diagnostics UI
- Support for every proprietary vendor/container format
- Real UFDR/UFDX generation
- Broad refactor of all ingest types beyond message parsing
- New investigative features unrelated to ingest adaptability

## Provenance & Audit Requirements
### Provenance
- All successfully parsed messages must continue to preserve:
  - SourceEvidenceItemId
  - SourceLocator
  - IngestModuleVersion / parser version where applicable
- Adaptability must not weaken evidence traceability.

### Audit
- Preserve existing ingest diagnostics and logging behavior.
- Add parser diagnostics as structured logs, not as user-action audit-log replacements.

## Performance & Resource Management Requirements
- Header/shape detection should be lightweight and bounded.
- Do not add expensive full-file scanning beyond what is needed for safe detection/mapping.
- Avoid blocking UI unnecessarily during ingest; continue to use the job queue / background parsing model.
- Keep alias matching and mapping logic maintainable and easy to extend.

## Implementation Notes / Edge Cases
- Treat message ingest as:
  1) detection
  2) mapping to canonical schema
  3) normalization
  4) persistence
- Suggested canonical fields:
  - MessageExternalId
  - ThreadExternalId
  - Platform
  - SentUtc
  - SenderDisplay
  - SenderIdentifier
  - RecipientDisplays
  - RecipientIdentifiers
  - Body
  - Direction
  - AttachmentCount
  - DeletedFlag
  - SourceEvidenceItemId
  - SourceLocator
  - ParserFamily
  - ParserVersion
  - ParseWarnings
- Suggested alias groups:
  - body/content/text/message
  - sender/from/source/author
  - recipient/to/destination/participants
  - timestamp/date/sent time/message date
  - thread/chat/conversation/chat id
- Prefer graceful partial parsing + warnings over rigid failure when enough information exists to normalize safely.
- Keep this ticket focused on adaptability and compatibility, not every vendor-specific parser edge case.

## Test Plan

### Integration tests (if applicable)

4. Confirm parsed messages appear in Search and other downstream features.
5. Review logs and confirm parser diagnostics clearly show matched fields and any warnings.
