# Ticket: T0041 - Gang documentation module v1

## Goal
Add a first-class Gang Documentation module under target/person profiles so investigators can create, maintain, review, and update formal gang documentation records tied to organizations, criteria, basis, approval status, and review lifecycle.

## Context
CaseGraph now has:
- organization registry and memberships
- navigable person and organization profile pages
- incident workspace and cross-reference flows
- cross-case identity foundations

The next logical step is a formal gang documentation workflow that lives under a target/person profile instead of as loose notes. The product needs a durable, reviewable structure for documenting gang affiliation, subgroup/set linkage, validation basis, status, and review requirements.

This should be a controlled documentation workflow, not just another free-text notes section.

## Scope
### In scope
- Add a **Gang Documentation** child record/workflow under target/person profiles.
- Each documentation record should support at least:
  - documentation id
  - target/global person link
  - linked organization
  - linked subgroup/set/clique if applicable
  - documentation status
  - approval status
  - reviewer
  - review due date
  - inactive/purge state
  - notes/summary
- Add support for **criteria/checklist entries** such as:
  - self-admission
  - tattoos / clothing / symbols
  - social media evidence
  - photos with documented members
  - message content / phone evidence
  - officer observation
  - jail / classification intel
  - confidential source
  - other documented basis
- Criteria entries should support at least:
  - criterion type
  - met/not met
  - basis summary
  - date observed
  - optional source note / reference summary
- Add workflow states such as:
  - draft
  - pending review
  - approved
  - inactive
  - purge review
- Add UI flows to:
  - create a gang documentation record from a target/person profile
  - edit/update documentation
  - manage criteria entries
  - update approval/review fields
  - browse documentation from the profile
- Add lightweight history/logging of documentation state changes where practical.
- Preserve separation between:
  - general profile data
  - formal gang documentation record

### Out of scope
- Full gang packet export
- Supervisor multi-stage review workflow beyond simple reviewer/approval fields
- Rivalry/territory modeling
- Organization hierarchy/pyramid visualization
- Automated scoring/recommendation engine
- Broad report builder integration

## Data Model Changes
- Add gang-documentation-focused data structures such as:
  - `GangDocumentationRecord`
  - `GangDocumentationCriterion`
  - optional `GangDocumentationStatusHistory`
- Exact names can be refined in implementation, but durable gang documentation persistence is required.

## Provenance & Audit Requirements
### Provenance
- Gang documentation basis should support source summaries or evidence references without weakening existing provenance behavior.
- Documentation should remain distinguishable from raw evidence-derived records.
- Existing View Source / Copy Citation behavior for linked evidence should remain intact where applicable.

### Audit
- Add structured logs and/or audit entries for:
  - documentation created
  - documentation updated
  - criterion added/updated/removed
  - approval/status changed
- Keep behavior consistent with existing CaseGraph audit/diagnostic patterns.

## Performance & Resource Management Requirements
- Documentation/profile loading should remain responsive.
- Criteria and documentation queries should be bounded and SQLite-safe.
- Avoid heavy cross-case scans when loading a single profile.
- Use deterministic resource disposal and safe async patterns.

## Implementation Notes / Edge Cases
- Keep v1 practical and investigator-focused.
- The module should support documenting:
  - full gang
  - subgroup/set/clique
  - associate vs member vs former vs suspected distinctions
- Documentation status should not overstate certainty.
- Suggested status fields:
  - draft
  - pending review
  - approved
  - inactive
  - purge review
- Criteria should support multiple entries per record.
- Basis summary can be lightweight in v1, but must be durable and editable.
- Reuse target/person/global-person and organization records rather than duplicating identity data.
- Keep the UI clean and readable:
  - no unlabeled blanks
  - clear empty states
  - clear separation between profile facts and formal gang documentation

1. Open a person/target profile.
2. Create a gang documentation record.
3. Link it to an organization and optional subgroup.
4. Add several criteria entries with basis summaries.
5. Set documentation status, reviewer, and review due date.
6. Save and refresh.
7. Confirm the documentation loads correctly and remains visible under the profile.
