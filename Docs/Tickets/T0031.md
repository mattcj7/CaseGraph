# Ticket: T0031 - Search readiness no-blocking and FTS maintenance manager

## Goal
Make the Search feature open immediately and prevent message FTS/index readiness work from blocking the UI or causing the app to appear hung. Search readiness must be fast when current, and visible/non-blocking when maintenance is needed.

## Context
After T0030, app startup and case-open performance improved significantly. However, opening Search and running a keyword query still hangs on real workspaces. Logs show the problem occurs during Search feature readiness at `WorkspaceMessageFtsEnsureStarted`, before the actual search query executes. This means the bottleneck is no longer app startup; it is now Search-specific FTS ensure/backfill behavior.

## Scope
### In scope
- Audit and stabilize Search feature readiness around message FTS/index ensure behavior.
- Separate:
  - fast readiness check
  - heavy repair/rebuild/backfill work
- Ensure Search tab opens immediately, even when FTS maintenance is needed.
- If FTS is current:
  - readiness should no-op quickly
  - search should execute immediately
- If FTS is stale/missing and repair is needed:
  - show visible “Preparing Search” / readiness state
  - keep UI responsive
  - run bounded maintenance in background or job-driven flow where safe
- Add explicit structured logging for:
  - FTS readiness check start/completion
  - FTS current/stale/missing status
  - backfill/rebuild decision
  - backfill/rebuild start/completion/failure
  - search query start/completion separate from readiness
- Add deterministic regression tests for:
  - Search opens without freezing
  - current FTS path is fast
  - stale/missing FTS path is visible and non-blocking

### Out of scope
- New search features or ranking logic
- New parsers or evidence types
- Broad search UI redesign
- Reworking unrelated feature readiness paths outside what is needed for a reusable pattern

## Acceptance Criteria (Testable)
- [ ] Search tab opens without freezing the UI.
- [ ] If message FTS is already current, search readiness completes quickly and the query runs immediately.
- [ ] If message FTS needs maintenance, the user sees a clear preparing/search-index state instead of a silent hang.
- [ ] FTS maintenance no longer causes the whole app to go “Not Responding”.
- [ ] Search query execution is clearly separated from readiness/maintenance in logs.
- [ ] Deterministic tests cover:
  - [ ] current FTS fast-path
  - [ ] stale/missing FTS visible non-blocking path
  - [ ] search remains responsive while readiness is in progress
- [ ] `dotnet test` passes.

## Provenance & Audit Requirements
### Provenance
- Must not weaken existing search provenance behavior.
- Search results must continue to support existing provenance/citation actions.

### Audit
- Preserve existing structured diagnostics and correlation ids.
- Search readiness and search query execution should be distinguishable in logs.

## Performance & Resource Management Requirements
- Search readiness must not block the UI thread.
- Heavy maintenance should be bounded, visible, and cancellable where safe.
- Avoid full rebuild/backfill on every Search open.
- Prefer quick no-op checks when FTS is already current.
- Dispose database resources deterministically.
- Avoid hidden infinite waits/retries.

## Implementation Notes / Edge Cases
- Likely hotspot:
  - message FTS ensure/backfill path currently reached during Search feature readiness
- Preferred behavior:
  - open Search immediately
  - if FTS is ready, search works immediately
  - if FTS is not ready, show a “Preparing Search” state and complete readiness without freezing the app
- Search query execution should not be conflated with maintenance in logs.
- If maintenance is deferred/backgrounded, clearly block only the specific search action if required, not the whole app.
- Keep this ticket focused on Search and reuse any generic readiness pattern only if it stays small and low-risk.

## Test Plan
### Unit tests
- current FTS readiness path returns quickly
- stale/missing FTS path transitions to preparing state
- logging labels readiness and query execution separately

### Integration tests (if applicable)
- Search open on initialized workspace does not trigger long blocking work
- Search open on stale/missing FTS workspace remains responsive and surfaces visible preparing state
- search executes after readiness completes

## How to Verify Manually
1. Launch the app on the real workspace.
2. Open a case.
3. Open Search and confirm the tab appears immediately.
4. Run a keyword query and confirm:
   - current-index path runs quickly, or
   - preparing/index-maintenance state is shown clearly without freezing the app
5. Review logs and confirm readiness vs query execution are clearly separated.
